# 最近 PB 项目工作流整理

本文档整理 2026-06-29 至 2026-07-01 期间 PB 项目协作形成的工作方式和业务流程。它只作为后续开发、验证、交付的参考文档，不替代模块记忆，也不改变任何业务代码、数据库或平台配置。

主要依据：

- `.codex/skills/pb-module-memory/references/modules/party-committee-work-plan-3.md`
- `.codex/skills/pb-module-memory/references/modules/multi-sub-table-excel-export.md`

## 1. 总览

这几天的工作沉淀出两条可复用流程：

1. **PB 模块协作工作流**：适用于以后新增模块、继续旧模块、修复问题、做内网交付。
2. **党委计划 3.0 业务工作流**：描述党委计划 3.0 的角色、任务状态、反馈链、附件、门户待办等业务流转。

共同原则：

- 先确认模块边界，再动代码。
- 新模块或大功能先建立模块记忆。
- 已有模块继续开发前先读模块记忆。
- 新建或修改表结构时必须检查 PB 强制审计字段。
- 前端改动必须保持 PB 平台隔离，不引入 CDN，不污染全局样式。
- 本地验证必须有明确结果，不能只凭页面观感判断完成。
- 内网交付前先更新模块记忆，再整理交付包、SQL 和人工平台步骤。

## 2. PB 模块协作工作流

### 2.1 进入条件

出现以下情况时，按本流程处理：

- 新增 PB 业务模块或较大功能。
- 继续已有模块开发。
- 修改低代码表单、视图、流程、菜单、权限、字典或平台配置。
- 修改业务表、SQL、Mapper、Service、JSP、JS、CSS。
- 准备内网部署、补丁包或交付清单。

### 2.2 标准步骤

1. **确认目标和边界**
   - 明确用户要解决的问题、最终使用对象、验收方式。
   - 判断是新模块、旧模块修正，还是纯交付工作。
   - 如果需求有多种解释，先列出解释并确认，不直接猜。

2. **建立或读取模块记忆**
   - 新模块或大功能：先询问是否创建模块记忆。
   - 已有模块：先在 `.codex/skills/pb-module-memory/references/modules/` 查找对应记忆。
   - 继续开发前必须读取已有业务规则、坑点、文件范围、表范围和交付说明。

3. **锁定隔离规则**
   - 明确本次允许修改的模块命名空间、页面目录、静态资源目录、表名前缀。
   - 明确禁止触碰的旧模块、旧表、旧菜单和旧接口。
   - 如果允许参考旧模块，只能作为参考或一次性导入来源，不能成为运行期依赖。

4. **设计最小实现**
   - 只实现当前需求需要的代码。
   - 不顺手重构无关模块。
   - 不添加未要求的通用能力、配置项或扩展点。
   - 前端优先使用项目已有 JSP、jQuery、EasyUI、layer、PB H5 组件和本地静态资源。

5. **实现和局部验证**
   - JS 改动后做语法检查。
   - Java 改动后用 JDK 8 编译相关类。
   - SQL 新表或改表后检查 PB 强制审计字段。
   - 前端资源改动后做 PB 前端冲突检查，重点确认无 CDN、无全局污染、资源路径正确。

6. **业务验证**
   - 用可重复脚本或明确测试数据覆盖主流程。
   - 测试数据必须带模块专用标记，例如 `DW3_*`，清理时只能清理这些标记数据。
   - 验证结果要记录具体命令、返回码、页面资源状态、接口返回或脚本标识。

7. **记录风险和决策**
   - 将重要业务规则、用户选择、否决过的方案、平台坑点写回模块记忆。
   - 不记录密码、token、本地私密信息。
   - 对未完成事项要写清楚原因和下一步。

8. **内网交付**
   - 用户明确要求部署或打包时才执行。
   - 先更新模块记忆，再准备交付包。
   - 交付包必须包含文件清单、SQL/配置迁移说明、人工平台步骤和禁止复制项。
   - 如果内网环境自行编译 Java，应提供源码包，不提供或不要求复制 `.class`。
   - 直接覆盖工程根目录的包，应让顶层目录就是 `src`、`WebRoot`、`sql` 等项目目录，避免误复制成 `project/files/src/...`。

### 2.3 固定检查项

开发前：

- 是否读过模块记忆。
- 是否明确本次允许修改的目录和表。
- 是否明确旧模块是否可参考、是否可运行期依赖。

开发中：

- 新表是否包含 `ID, CREATED_BY, CREATION_DATE, LAST_UPDATED_BY, LAST_UPDATE_DATE, LAST_UPDATE_IP, VERSION, ORG_IDENTITY`。
- 前端是否只用本地资源。
- 是否避免影响 PB 全局样式和平台组件。

完成前：

- JS 语法检查通过。
- Java 相关类编译通过。
- SQL 审计字段检查通过。
- 前端冲突检查无阻断问题。
- 主业务脚本或人工验证路径有明确结果。
- 模块记忆已记录重要变更、风险和交付说明。

## 3. 党委计划 3.0 业务工作流

### 3.1 模块边界

党委计划 3.0 是独立模块：

- 运行期代码只使用 `dwworkplan3`。
- 运行期数据只使用 `DYN_DW_PLAN3_*`。
- 不调用旧 `avicit/pb/dwworkplan` Controller。
- 不读写旧 `DYN_DW_PLAN_*` 表。
- 旧模块页面、交互、人员树结构可以参考，也可以一次性导入数据，但不能作为运行期依赖。
- 不使用 BPM 流程。
- 门户待办通过 `PB_PORTAL_BUSINESS_TODO` 补齐。

主要入口：

- `platform/avicit/pb/dwworkplan3/dwWorkPlan3Controller/toIndex`

主要表：

- `DYN_DW_PLAN3_BATCH`
- `DYN_DW_PLAN3_PERSON_TREE`
- `DYN_DW_PLAN3_TASK`
- `DYN_DW_PLAN3_FEEDBACK`
- `DYN_DW_PLAN3_ATTACHMENT`
- `PB_PORTAL_BUSINESS_TODO`

### 3.2 角色和可见范围

- **党委计划下发者**：可创建季度批次、根任务，查看全部任务、人员树和统计。
- **部长**：查看自己人员节点及下级子树，接收党委任务后向室主任下发，或在无子任务时直接反馈。
- **室主任**：查看自己节点及下级子树，接收部长任务后向科员下发，或在无子任务时直接反馈。
- **科员**：只处理自己的工作计划任务，不能继续下发。

人员树规则：

- 部门和科室节点允许绑定多个平台用户。
- 党委和科员节点保持单用户绑定。
- 同一个平台用户不能同时绑定到不同层级的启用人员节点，避免一个人既是科员又是部长时权限判断混乱。
- 人员树表格当前只展示节点名称，用户和排序在右侧维护表单中编辑。

### 3.3 任务下发流程

1. 党委创建季度批次。
2. 党委创建根任务。
3. 党委可保存草稿，也可直接下发给部长。
4. 部长接收任务后，下发给室主任。
5. 室主任接收任务后，下发给科员。
6. 科员接收后完成并提交反馈。
7. 上级逐级确认或退回。

关键规则：

- 季度批次只是分组，不限制任务数量。
- 同一个任务只要已经下发过一次，就不能再次下发。
- 科员不能继续下发。
- 未接收的下级任务允许发送方撤回。
- 已接收的下级任务不能撤回。
- 删除只允许党委计划下发者操作，删除时移除选中任务子树及相关 3.0 反馈和附件记录。

### 3.4 状态流转

常见状态含义：

- `DRAFT`：草稿，未下发。
- `TODO`：已下发，接收人待接收。
- `DOING`：接收人已接收，处理中。
- `WAIT_CHILD`：已有下级任务，等待下级完成。
- `PENDING_CONFIRM`：下级已提交反馈，等待上级确认。
- `RETURNED`：反馈被退回，接收人需重新处理。
- `COMPLETED`：已确认完成。

流转规则：

- 下发后生成接收人的待办。
- 接收后保持或刷新接收人的处理待办。
- 提交反馈后关闭处理待办，生成上级确认待办。
- 退回后关闭确认待办，重新生成接收人的处理待办。
- 确认后关闭相关待办。
- 撤回、删除、批量删除时关闭相关业务待办。

### 3.5 反馈链流程

反馈从下级向上级流转：

1. 科员提交反馈。
2. 室主任确认或退回。
3. 室主任确认后，形成上一级反馈内容。
4. 部长确认或退回。
5. 部长确认后，继续向党委反馈。
6. 党委最终确认或退回。

反馈规则：

- 退回必须填写退回原因。
- 被退回的反馈历史必须保留。
- 重新提交时创建新的待确认反馈，不覆盖旧退回记录。
- 反馈链展示顺序按业务流向从最下级开始向上编号。
- 反馈内容必须清晰展示，不能被挤压在窄列中。

### 3.6 附件链路

附件使用 PB 平台 `fileupload/uploaderExt`：

- 任务附件逻辑 ID：`dwTaskAttachment`
- 反馈附件逻辑 ID：`dwFeedbackAttachment`
- 上传视图和只读视图必须使用稳定逻辑 ID，不能使用 DOM 容器 ID 当业务元素 ID。
- 下级用户必须能看到直接父任务附件。
- 科员必须能看到室主任下发时携带的附件。
- 反馈附件在反馈链中以紧凑按钮或列表入口展示，不在表格单元格内嵌完整上传控件。
- 只读附件弹窗允许下载，禁用新增、删除、预览和保存网盘。

### 3.7 门户待办流程

党委计划 3.0 不走 BPM，因此门户待办使用通用业务待办表：

- 表：`PB_PORTAL_BUSINESS_TODO`
- 门户入口：`/ims/oa/todo/uniontask/pendingWork`
- 不新增 `/pendWork`。

待办规则：

- `TODO`、`DOING`、`RETURNED`：给接收人创建 `HANDLE` 待办。
- `PENDING_CONFIRM`：给发送方创建 `CONFIRM` 待办。
- `DRAFT`、`WAIT_CHILD`、`COMPLETED`：不保留活跃业务待办。
- 表不存在时，业务待办写入服务应安全跳过，不影响原 BPM 待办。

### 3.8 查询、统计和界面规则

- 工作计划筛选不使用手动搜索按钮。
- 年季、状态变化后立即刷新。
- 关键词输入使用短延迟自动刷新。
- 年季下拉只显示当前可见范围内有任务的批次。
- 党委看全部统计；部长和室主任看自己节点及下级；科员不进入统计。
- 人员树层级标签按党委、部门、科室、科员区分颜色。
- KPI 卡片突出数字和状态，不使用过重的外框。

## 4. Excel 导出模块交付经验

多子表 Excel 导出配置模块沉淀出一条内网交付经验：

- 管理页使用 JSP + jQuery/jqGrid/layer。
- 后端使用 Spring MVC + JdbcTemplate。
- 导出接口按 `code + id` 读取已保存配置，不接受请求直接传入任意表名或字段名。
- 表名和字段名必须通过数据库元数据校验。
- 配置表必须包含 PB 强制审计字段。
- 内网如果由测试环境自行编译 Java，应提供源码包。
- 源码包最好按工程根目录组织，顶层直接包含 `src`、`WebRoot`、`sql`，避免复制路径错误。
- SQL 需要说明：新环境执行建表脚本；旧环境确认补齐新增字段。
- 菜单 URL 和 Tomcat 重启属于人工平台步骤，必须写入交付说明。

## 5. 后续使用方式

以后继续 PB 项目时：

1. 先判断任务属于哪个模块。
2. 先读对应模块记忆。
3. 再按本文档的协作工作流推进。
4. 涉及党委计划 3.0 时，额外检查本文档第 3 节的业务规则。
5. 涉及内网交付时，参考第 4 节的交付经验。
